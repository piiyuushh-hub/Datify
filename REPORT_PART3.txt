11.5 Key Frontend Features

TOAST NOTIFICATIONS:
  Replaces browser alerts with sleek, auto-dismissing toast messages.
  Types: success (green, checkmark) or error (orange, warning icon).
  Auto-dismisses after 3 seconds with CSS slide animation.

THEME TOGGLE:
  Light/Dark mode toggle persisted in localStorage.
  Applies data-theme="light" attribute to <html> element.
  CSS variables switch entire color palette automatically.

SKELETON LOADING:
  While prediction history loads, animated skeleton placeholder cards
  are shown instead of a blank space — improves perceived performance.

CSV EXPORT:
  Exports prediction history as a .csv file with columns:
  Timestamp, Telecom Partner, Gender, Age, City, Salary, Predicted Data (GB)
  Uses Blob API and a hidden anchor element for download.

RETRY LOGIC:
  Prediction requests retry up to 4 times if the server returns 502/504
  (meaning FastAPI is still warming up). Useful for cold-start scenarios.

CHAPTER 12: BACKEND API ARCHITECTURE (NODE.JS)
================================================================================

12.1 server.js — The Entry Point
----------------------------------
The Node.js Express server (port 5000) does four things:
  1. Spawns the FastAPI child process automatically on startup
  2. Connects to MongoDB Atlas
  3. Registers all API routes
  4. Serves the frontend static files

FastAPI Child Process Spawning:
  spawn('python', ['-m', 'uvicorn', 'app:app', '--host', '127.0.0.1',
                   '--port', '8001', '--app-dir', '../backend'])
  - FastAPI runs internally on 127.0.0.1:8001 (not exposed externally)
  - If FastAPI crashes, it auto-restarts after 3 seconds
  - Node.js logs FastAPI stdout/stderr for debugging

12.2 Route Summary
-------------------

  ROUTE                     METHOD    AUTH?   PURPOSE
  ──────────────────────────────────────────────────────────────────
  /api/auth/register        POST      No      Create new account
  /api/auth/login           POST      No      Login, returns JWT
  /api/predict              POST      Yes     Run ML prediction
  /api/history              GET       Yes     Get last 10 predictions
  /api/history/clear        DELETE    Yes     Delete all history
  /api/history/analytics    GET       Yes     Get chart aggregations
  /api/history/:id          DELETE    Yes     Delete one prediction
  /* (catch-all)            GET       No      Serve frontend SPA

12.3 predict.js Route Detail
------------------------------
  1. JWT middleware checks Authorization header
  2. Extracts 15 input fields from request body
  3. POSTs to http://127.0.0.1:8001/predict (FastAPI)
  4. Receives { predicted_data_used_gb, explanations }
  5. Creates new Prediction document in MongoDB
  6. Returns prediction + explanations + predictionId to browser

  Error handling:
  - ECONNREFUSED: FastAPI still booting → "AI engine warming up" message
  - 502/503: Handled by frontend retry logic

12.4 history.js Analytics Pipeline
-------------------------------------
Uses MongoDB aggregation framework for chart data:

  CITY AGGREGATION:
    db.predictions.aggregate([
      { $match: { userId: ObjectId(userId) } },
      { $group: { _id: "$inputs.city",
                  avgData: { $avg: "$predicted_data_used_gb" } } },
      { $sort: { _id: 1 } }
    ])

  SALARY BUCKET AGGREGATION:
    db.predictions.aggregate([
      { $match: { userId: ObjectId(userId) } },
      { $bucket: {
          groupBy: "$inputs.estimated_salary",
          boundaries: [0, 40000, 60000, 80000, 100000, 120000, 150000],
          default: "150k+",
          output: {
            avgData: { $avg: "$predicted_data_used_gb" },
            count: { $sum: 1 }
          }
      }}
    ])

CHAPTER 13: DATABASE SCHEMA & MONGODB DESIGN
================================================================================

13.1 Overview
--------------
MongoDB Atlas (cloud) is used as the database. Mongoose ODM provides
schema validation and an object-oriented interface in Node.js.

Three collections exist:

13.2 Collection 1 — Users
---------------------------
Schema (User.js):
  {
    username:  String (required, unique, trimmed),
    password:  String (required, bcrypt-hashed),
    createdAt: Date (default: now)
  }

  Behaviour:
  - Pre-save hook automatically hashes the password with bcrypt (10 rounds)
  - comparePassword() method does bcrypt comparison for login
  - Password is NEVER stored in plain text

13.3 Collection 2 — Predictions
---------------------------------
Schema (Prediction.js):
  {
    userId:                 ObjectId (ref: User, required),
    inputs: {
      age, gender, telecom_partner, city, num_dependents,
      estimated_salary, calls_made, sms_sent, streaming_hours,
      gaming_hours, social_media_hours, wifi_ratio,
      weekend_usage_ratio, background_data_mb, auto_update
    },
    predicted_data_used_gb: Number (required),
    timestamp:              Date (default: now)
  }

  - Every prediction is permanently saved with ALL 15 input values
  - Linked to a user via userId (foreign key pattern)
  - timestamp enables chronological history ordering

13.4 Collection 3 — Profiles
------------------------------
Schema (Profile.js):
  {
    userId:      ObjectId (ref: User),
    profileName: String (required),
    predicted_gb: Number (required),
    inputs: { all 15 fields },
    createdAt:   Date (default: now)
  }

  Allows users to save named profiles (e.g., "My Work Phone", "Home Plan")
  with all their input values and the corresponding prediction.

CHAPTER 14: SECURITY IMPLEMENTATION
================================================================================

14.1 Password Security
-----------------------
  Library: bcryptjs
  Salt rounds: 10 (industry standard)

  Registration flow:
    1. User submits username + password
    2. Frontend validates password strength:
       Must have: 1 uppercase, 1 lowercase, 1 number, 1 special character
    3. Node.js double-validates with same regex
    4. bcrypt.genSalt(10) generates a unique random salt
    5. bcrypt.hash(password, salt) creates irreversible hash
    6. Only the hash is stored in MongoDB — NEVER the original password

  Login flow:
    1. User submits password
    2. bcrypt.compare(inputPassword, storedHash) — timing-safe comparison
    3. Returns true/false without exposing the original password

14.2 JWT Authentication
------------------------
  Library: jsonwebtoken
  Token validity: 24 hours
  Secret: Stored in .env file as JWT_SECRET (never hardcoded)

  Token structure:
    Header:    { alg: "HS256", typ: "JWT" }
    Payload:   { userId: "MongoDB ObjectId", iat: timestamp, exp: timestamp }
    Signature: HMACSHA256(base64(header) + "." + base64(payload), JWT_SECRET)

  Token flow:
    1. User logs in successfully
    2. Server issues signed JWT
    3. Frontend stores JWT in localStorage
    4. Every protected API call sends: Authorization: Bearer <token>
    5. auth.js middleware verifies signature and expiry
    6. Decoded userId attached to req.user for route handlers

14.3 CORS Policy
-----------------
  FastAPI: allow_origins=["*"] — accepts requests from any origin
  Node.js: cors() middleware — allows cross-origin requests during development

14.4 Environment Variables (.env)
-----------------------------------
  Sensitive configuration stored in node_backend/.env:
    MONGODB_URI=mongodb+srv://...  (MongoDB Atlas connection string)
    JWT_SECRET=...                 (Random secret for JWT signing)
    PORT=5000                      (Node.js server port)

  This file is NEVER committed to version control (.gitignore).

CHAPTER 16: LIMITATIONS & FUTURE IMPROVEMENTS
================================================================================

16.1 Current Limitations

DATASET LIMITATIONS:
  - The dataset is synthetic (simulated), not from real telecom operators.
    Real-world data from Jio/Airtel would produce even more accurate models.
  - The dataset covers only 6 Indian cities. Smaller cities and rural areas
    are not represented, limiting geographic generalization.
  - Only 4 telecom operators are included. MVNOs and newer operators are absent.

MODEL LIMITATIONS:
  - n_estimators=30 was chosen for training speed. A production system would
    use 100-500 trees for higher accuracy.
  - No hyperparameter tuning (GridSearchCV or RandomizedSearchCV) was performed.
    Systematic tuning could push R² above 0.98.
  - The weekend_usage_ratio feature required a manual heuristic because its
    learned importance in the model was not strong enough on its own.
  - The model does not use time-series data (monthly trends, seasonal effects).

TECHNICAL LIMITATIONS:
  - Running FastAPI as a subprocess of Node.js is not ideal for production at
    scale. Docker containerization would be the proper approach.
  - No rate limiting on the prediction API (vulnerable to abuse).
  - No input sanitization beyond basic type validation.
  - JWT tokens cannot be revoked (no token blacklist / refresh token system).
  - The frontend is not a proper SPA framework (React/Vue), making complex
    state management more difficult.

16.2 Recommended Future Improvements

ML IMPROVEMENTS:
  1. Use XGBoost or LightGBM — faster, often more accurate than sklearn GB
  2. Implement GridSearchCV for systematic hyperparameter optimization
  3. Add SHAP (SHapley Additive exPlanations) for proper model-level XAI
     instead of the current rule-based heuristic approach
  4. Train on real anonymized telecom data for production accuracy
  5. Add time-series features (last month's usage, rolling 3-month average)
  6. Build a model retraining pipeline that updates model.pkl monthly

API IMPROVEMENTS:
  1. Migrate to Docker + Docker Compose for proper service isolation
  2. Add Redis caching for repeated identical prediction requests
  3. Implement rate limiting (express-rate-limit)
  4. Add comprehensive request logging with Winston or Morgan
  5. Implement refresh tokens for longer sessions without security risk

FRONTEND IMPROVEMENTS:
  1. Migrate to React.js for proper component-based architecture
  2. Add a "Plan Recommender" — after predicting, suggest the best data plan
  3. Add multi-language support (Hindi, Tamil, Telugu) for wider reach
  4. Implement Progressive Web App (PWA) for offline capability
  5. Add email-based password reset functionality

DATA IMPROVEMENTS:
  1. Integrate real-time 4G/5G tower coverage data by city
  2. Add device type as a feature (iPhone vs Android — different usage patterns)
  3. Include app-specific breakdown (Netflix vs YouTube vs Instagram usage)

17.1 Project Summary
---------------------
Datify AI is a complete, end-to-end machine learning web application that
successfully demonstrates the full pipeline of a modern AI product:

  DATA → CLEANING → TRAINING → EVALUATION → DEPLOYMENT → USER INTERFACE

The project proves that a well-engineered machine learning solution can achieve
remarkable accuracy (R² = 0.97) on a realistic, complex, multi-feature dataset,
and that this intelligence can be packaged into a beautiful, accessible web
application that non-technical users can interact with intuitively.

17.2 Key Achievements
----------------------

  1. MODEL ACCURACY:
     The Random Forest Regressor achieved R² = 0.97 and RMSE = 0.50 GB,
     meaning it can predict a customer's monthly data usage to within
     approximately half a gigabyte — a remarkably precise result.

  2. FULL-STACK INTEGRATION:
     Successfully integrated Python ML, FastAPI, Node.js, MongoDB, and a
     vanilla JS frontend into a cohesive, working application with proper
     authentication and data persistence.

  3. EXPLAINABLE AI:
     The XAI system makes the model's decisions transparent, providing users
     with plain-English explanations of which factors drove their prediction.
     This is critical for user trust in AI systems.

  4. PRODUCTION PATTERNS:
     Implemented industry-standard practices: JWT authentication, bcrypt
     hashing, environment variables, ML pipelines, model serialization,
     automatic service restart, and MongoDB aggregation for analytics.

  5. UX EXCELLENCE:
     Features like Quick Fill Profiles, Baseline Comparison, animated results,
     toast notifications, CSV export, and live charts make the application
     genuinely useful and engaging.
