CHAPTER 6: MACHINE LEARNING MODELS — THEORY & CONCEPTS
================================================================================

6.1 The Problem Type: Regression
----------------------------------
This project solves a SUPERVISED REGRESSION problem.

  SUPERVISED: The model learns from labeled examples (we provide both the
  input features AND the correct answer — monthly_data_usage).

  REGRESSION: The output is a continuous number (e.g., 4.72 GB), not a
  category. This is different from classification (e.g., "high" vs "low").

  We trained and compared THREE regression algorithms:
    1. Linear Regression
    2. Random Forest Regressor
    3. Gradient Boosting Regressor

─────────────────────────────────────────────────────────────────────────────
6.2 MODEL 1: Multiple Linear Regression
─────────────────────────────────────────────────────────────────────────────

CONCEPT:
  Linear Regression is the most fundamental supervised learning algorithm.
  It assumes that the relationship between input features and the target
  variable can be expressed as a STRAIGHT LINE (in multi-dimensional space,
  this becomes a hyperplane).

MATHEMATICAL FORMULA:
  y = b0 + b1*x1 + b2*x2 + b3*x3 + ... + bn*xn

  Where:
    y  = predicted monthly_data_usage (in GB)
    b0 = intercept (bias term)
    b1, b2, ..., bn = learned coefficients (weights) for each feature
    x1, x2, ..., xn = input feature values

  Example:
    monthly_data_usage = 1.5 + (0.8 * streaming_hours)
                              + (-3.2 * wifi_ratio)
                              + (0.4 * gaming_hours)
                              + (0.0001 * estimated_salary)
                              + ...

HOW IT LEARNS:
  The algorithm finds the values of b0, b1, ..., bn that MINIMIZE the
  total squared error between its predictions and the actual values.
  This method is called "Ordinary Least Squares" (OLS).

  Objective: Minimize Σ(y_actual - y_predicted)²

PREPROCESSING REQUIRED:
  - StandardScaler is essential because LinearRegression is sensitive to
    the scale of features. Without scaling, high-magnitude features
    dominate the coefficient calculation.

STRENGTHS:
  + Extremely fast to train and predict
  + Highly interpretable (you can see each coefficient)
  + Works well when the data has a truly linear relationship
  + Low risk of overfitting

WEAKNESSES:
  - CANNOT capture non-linear relationships (e.g., the interaction between
    wifi_ratio and streaming_hours is complex and non-linear)
  - Assumes independence of features (multicollinearity causes issues)
  - Performs poorly on this telecom dataset because data usage behaviour
    is fundamentally non-linear

RESULT ON THIS PROJECT:
  R² Score: ~0.72 (explains only 72% of variance)
  RMSE:     ~1.54 GB
  VERDICT:  Not selected — too simplistic for this problem

─────────────────────────────────────────────────────────────────────────────
6.3 MODEL 2: Random Forest Regressor
─────────────────────────────────────────────────────────────────────────────

CONCEPT:
  Random Forest is an ENSEMBLE LEARNING method. Instead of building one
  decision tree, it builds MANY decision trees (a "forest") and averages
  their predictions. Each tree is trained on a random subset of the data
  and a random subset of features.

  The core principle is "Wisdom of the Crowd" — many weak/average predictors
  together outperform a single strong predictor.

WHAT IS A DECISION TREE?
  A Decision Tree splits the data using a series of if/else rules:
    IF streaming_hours > 3.0:
        IF wifi_ratio < 0.3:
            PREDICT 12.5 GB
        ELSE:
            PREDICT 6.2 GB
    ELSE:
        IF social_media_hours > 4.0:
            PREDICT 5.1 GB
        ELSE:
            PREDICT 3.2 GB

  Each tree learns different rules because it sees different random data.

HOW RANDOM FOREST WORKS:
  1. BOOTSTRAP SAMPLING: For each of the n_estimators trees, a random
     sample of training rows is drawn WITH REPLACEMENT (bagging).
  2. FEATURE RANDOMNESS: At each split in each tree, only a random subset
     of features is considered. This decorrelates the trees.
  3. AGGREGATION: For regression, the final prediction = MEAN of all tree
     predictions. For n=30 trees, the model averages 30 predictions.

  prediction = (tree1_pred + tree2_pred + ... + tree30_pred) / 30

PARAMETERS USED IN THIS PROJECT:
  RandomForestRegressor(
      n_estimators = 30,    # Number of trees in the forest
      max_depth    = 8,     # Maximum depth of each tree (prevents overfitting)
      random_state = 42,    # Reproducibility seed
      n_jobs       = -1     # Use ALL CPU cores for parallel training
  )

  n_estimators=30: Chosen to balance accuracy vs. training speed.
    More trees = better accuracy but slower training. 30 is practical
    for a local machine.

  max_depth=8: Each tree is allowed up to 8 levels of splits. Without
    a max_depth, trees can grow until every training sample has its own
    leaf (perfect training accuracy but catastrophic test performance —
    this is called OVERFITTING). Depth=8 provides enough complexity to
    capture real patterns while still generalising.

STRENGTHS:
  + Handles non-linear relationships extremely well
  + Robust to outliers (averaging across many trees reduces their impact)
  + Automatically handles feature interactions
  + Built-in feature importance scores
  + Parallelizable — can use all CPU cores (n_jobs=-1)
  + Very low risk of overfitting due to averaging and depth limit

WEAKNESSES:
  - Slower to predict than a single tree
  - Larger model file size (30 trees vs. 1)
  - Less interpretable than linear regression (black-box nature)
  - Requires more memory than simpler models

RESULT ON THIS PROJECT:
  R² Score: ~0.97 (explains 97% of variance in data usage)
  RMSE:     ~0.50 GB
  VERDICT:  *** SELECTED AS THE PRODUCTION MODEL ***

─────────────────────────────────────────────────────────────────────────────
6.4 MODEL 3: Gradient Boosting Regressor
─────────────────────────────────────────────────────────────────────────────

CONCEPT:
  Gradient Boosting is another ensemble method, but it builds trees
  SEQUENTIALLY rather than in parallel like Random Forest. Each new tree
  is specifically trained to correct the ERRORS (residuals) of the
  combined ensemble of all previous trees.

  The word "Gradient" comes from the fact that the algorithm uses
  gradient descent to minimise a loss function at each step.

HOW IT WORKS (Step by Step):
  Step 1: Start with a simple prediction (the mean of y).
  Step 2: Calculate the residuals (actual - predicted).
  Step 3: Train a new tree to predict those residuals.
  Step 4: Add this tree to the ensemble with a learning_rate scaling factor.
  Step 5: Recalculate residuals.
  Step 6: Repeat for n_estimators iterations.

  Mathematical view:
    F0(x) = mean(y)
    F1(x) = F0(x) + lr * tree1(x)  [where tree1 predicts F0's residuals]
    F2(x) = F1(x) + lr * tree2(x)  [where tree2 predicts F1's residuals]
    ...
    Fm(x) = F(m-1)(x) + lr * tree_m(x)

  Each tree "boosts" the ensemble's performance by focusing on the cases
  that previous trees got wrong.

PARAMETERS USED IN THIS PROJECT:
  GradientBoostingRegressor(
      n_estimators = 30,   # Number of sequential trees
      max_depth    = 5,    # Shallower trees (5 vs 8) — typical for GB
      random_state = 42
  )

  max_depth=5: Gradient Boosting trees are intentionally shallow ("stumps")
  because the boosting process itself provides complexity. Deeper trees
  in GB lead to overfitting faster.

STRENGTHS:
  + Often achieves the highest accuracy of any non-deep-learning method
  + Excellent at capturing complex patterns
  + Very flexible — works well for many problem types

WEAKNESSES:
  - SLOW to train (sequential, cannot be parallelized like Random Forest)
  - More sensitive to hyperparameters (learning_rate, n_estimators, depth)
  - More prone to overfitting than Random Forest
  - Harder to tune correctly

RESULT ON THIS PROJECT:
  R² Score: ~0.95 (explains 95% of variance)
  RMSE:     ~0.65 GB
  VERDICT:  Good model, but slightly outperformed by Random Forest here.

CHAPTER 7: MODEL TRAINING & EVALUATION
================================================================================

7.1 The Training Loop
----------------------
Each model is wrapped in a scikit-learn Pipeline (see Chapter 9) and trained
on the same X_train / y_train data:

  for name, model in models.items():
      pipeline = Pipeline(steps=[
          ('preprocessor', preprocessor),  # Scale + Encode
          ('regressor', model)              # Train the ML model
      ])
      pipeline.fit(X_train, y_train)        # FIT on training data
      y_pred = pipeline.predict(X_test)     # PREDICT on test data
      rmse = root_mean_squared_error(y_test, y_pred)
      r2   = r2_score(y_test, y_pred)

7.2 Evaluation Metrics
------------------------

METRIC 1: R² Score (Coefficient of Determination)
  Formula: R² = 1 - (SS_residuals / SS_total)
    SS_residuals = Σ(y_actual - y_predicted)²
    SS_total     = Σ(y_actual - y_mean)²

  INTERPRETATION:
    R² = 1.0  → Perfect model (predicts every value exactly)
    R² = 0.97 → The model explains 97% of the variation in data usage
    R² = 0.0  → The model is no better than just predicting the mean
    R² < 0    → The model is WORSE than predicting the mean (bad model)

  R² is the primary metric used to select the best model.

METRIC 2: RMSE — Root Mean Squared Error
  Formula: RMSE = √[ (1/n) * Σ(y_actual - y_predicted)² ]

  INTERPRETATION:
    RMSE is expressed in the SAME UNITS as the target variable (GB).
    An RMSE of 0.50 GB means: on average, the model's predictions are
    off by approximately 0.50 GB from the actual value.

    Lower RMSE = Better model.

    For context: if a customer actually uses 5.0 GB and the model predicts
    5.5 GB, the error is 0.5 GB — which is highly acceptable.

7.4 Why R² of 0.97 is Excellent
---------------------------------
In real-world machine learning on messy, human behavioural data, achieving
R² above 0.90 is considered excellent. An R² of 0.97 means:
  - The model captures 97% of the pattern in how people use data
  - Only 3% of variance is unexplained (attributable to truly random
    individual behaviour that no model could predict)
  - The model generalises very well to unseen customers

8.1 Selection Logic
--------------------
The selection is entirely automated. After evaluating all three models,
the code tracks which model has the highest R² score:

  best_model = None
  best_r2    = -infinity

  for name, model in models.items():
      ...evaluate...
      if r2 > best_r2:
          best_r2        = r2
          best_model     = pipeline
          best_model_name = name

  # Save the winner
  joblib.dump(best_model, 'backend/model.pkl')

8.2 Why Random Forest Was Selected
------------------------------------
Random Forest won because:

  1. DATA IS NON-LINEAR: Mobile data usage depends on complex interactions
     between features. For example, a customer who streams 4 hours/day on
     Wi-Fi uses far less cellular data than one streaming 4 hours on cellular.
     This INTERACTION between streaming_hours and wifi_ratio is non-linear
     and easily captured by tree-based models.

  2. FEATURE DIVERSITY: With 15 diverse features (categorical + numerical,
     ranging from salary to gaming hours), Random Forest naturally handles
     this mixed-type dataset without requiring manual engineering.

  3. OUTLIER RESISTANCE: Some customers have extreme values (e.g., someone
     who games 8 hours/day). The averaging nature of Random Forest reduces
     the impact of such outliers on the final prediction.

  4. PARALLELISM: n_jobs=-1 allowed all CPU cores to train trees
     simultaneously, making it faster to train than Gradient Boosting.

  5. EMPIRICAL PERFORMANCE: On the actual test set (10,000 rows of unseen
     customers), it simply achieved the highest R² (0.97) and lowest RMSE.

8.3 Model Persistence
----------------------
The best model is serialized (saved to disk) using Python's joblib library:

  joblib.dump(best_model, 'backend/model.pkl')

  model.pkl size: ~4.6 KB (compact because only 30 shallow trees)

  To load and use in production:
  model_pipeline = joblib.load('backend/model.pkl')
  prediction = model_pipeline.predict(new_data_dataframe)

  joblib is preferred over Python's built-in pickle for:
    - More efficient serialization of large NumPy arrays
    - Better compression of scikit-learn objects
    - Faster load/dump operations

CHAPTER 9: THE SCIKIT-LEARN PIPELINE
================================================================================

9.1 What is a Pipeline?
------------------------
A scikit-learn Pipeline chains multiple processing steps together into a
single object. This is critical for correct, production-safe ML.

  pipeline = Pipeline(steps=[
      ('preprocessor', preprocessor),  # Step 1: Scale + OneHotEncode
      ('regressor', model)              # Step 2: Train/predict with ML model
  ])

  When pipeline.fit(X_train, y_train) is called:
    1. preprocessor.fit_transform(X_train) is run
    2. regressor.fit(scaled_encoded_X_train, y_train) is run

  When pipeline.predict(X_new) is called:
    1. preprocessor.transform(X_new) is run (uses parameters from step 1)
    2. regressor.predict(scaled_encoded_X_new) is run

9.2 Why Pipelines Are Essential — The Data Leakage Problem
-----------------------------------------------------------
WITHOUT a pipeline (the WRONG way):
  scaler.fit_transform(X_train)  # Fits scaler ON TRAINING DATA
  scaler.transform(X_test)       # Fine — applies training stats to test
  
  BUT if someone accidentally calls:
  scaler.fit_transform(X)        # Fits scaler on ALL data including test!
  This causes DATA LEAKAGE — the model has "seen" the test data, making
  evaluation results dishonestly optimistic.

WITH a pipeline (the CORRECT way):
  The pipeline guarantees the preprocessor is ONLY fitted on training data.
  The test data is ONLY transformed (not used to fit), which is statistically
  honest and prevents leakage.

9.3 The ColumnTransformer
--------------------------
The preprocessor is a ColumnTransformer that applies DIFFERENT transformations
to DIFFERENT columns in parallel:

  preprocessor = ColumnTransformer(transformers=[
      ('num', StandardScaler(), numerical_cols),          # Scale numbers
      ('cat', OneHotEncoder(handle_unknown='ignore'), categorical_cols)  # Encode text
  ])

  After transformation, the output is a single NumPy matrix where:
    - Numerical columns are standardized (mean=0, std=1)
    - Categorical columns are expanded into binary one-hot columns
    - All columns are now numerical — ready for the ML model

  Example: A row like [age=25, salary=80000, city='Mumbai', ...] becomes:
    [−1.2, 0.8, −0.3, ..., 1, 0, 0, 0, 1, 0, ...]
     ↑ scaled numbers         ↑ one-hot for city  ↑ one-hot for gender

CHAPTER 10: THE PREDICTION ENGINE & EXPLAINABLE AI (XAI)

10.1 How Predictions Flow in Production
-----------------------------------------

STEP 1: User fills the 15-field form on the dashboard and clicks "Predict"
STEP 2: JavaScript collects all values and sends POST /api/predict with JWT
STEP 3: Node.js middleware verifies the JWT token
STEP 4: Node.js forwards the JSON payload to FastAPI at http://127.0.0.1:8001/predict
STEP 5: FastAPI receives the 15 features
STEP 6: FastAPI converts them to a pandas DataFrame (single row)
STEP 7: model_pipeline.predict(df) is called — returns a float (GB)
STEP 8: Weekend ratio heuristic modifier is applied
STEP 9: XAI explanations are generated
STEP 10: Response returned to Node.js
STEP 11: Node.js saves prediction to MongoDB (Prediction collection)
STEP 12: Node.js returns result + explanations to browser
STEP 13: JavaScript animates the result number and renders XAI cards

10.2 The Weekend Ratio Heuristic Modifier
------------------------------------------
An important post-processing heuristic is applied to make the weekend_usage_ratio
feature have a more visible, intuitive impact on the final prediction:

  weekend_ratio    = req.weekend_usage_ratio
  weekend_modifier = 1.0 + (weekend_ratio - 0.28) * 0.4
  final_prediction = base_ml_prediction * weekend_modifier

  LOGIC:
    - 0.28 is the natural baseline (2 weekend days out of 7 = 28.6%)
    - If weekend_ratio = 0.28: modifier = 1.0 (no change)
    - If weekend_ratio = 0.60: modifier = 1.0 + (0.32 * 0.4) = 1.128
      → 12.8% MORE data than base prediction
    - If weekend_ratio = 0.10: modifier = 1.0 + (-0.18 * 0.4) = 0.928
      → 7.2% LESS data than base prediction

  WHY THIS HEURISTIC? The ML model was trained on a dataset where the
  weekend_usage_ratio column existed but its learned weight may not have
  been strong enough to produce visibly different outputs when a user
  moves the slider. This heuristic ensures the UI is responsive and
  educational — users can see how weekend binge behaviour affects their bill.

10.3 Explainable AI (XAI) — How It Works
------------------------------------------
After the ML model produces a numerical prediction, the system generates
human-readable explanations using rule-based heuristics. This is a form
of Explainable AI (XAI) — making AI decisions transparent to users.

The XAI engine checks each key feature and generates explanations:

  IF streaming_hours > 3.0:
      "High streaming hours significantly increased your predicted data consumption."
      Effect: POSITIVE (red indicator — increases usage)

  ELIF streaming_hours < 1.0:
      "Low streaming hours kept your data consumption low."
      Effect: NEGATIVE (green indicator — decreases usage)

  IF wifi_ratio > 0.7:
      "A high Wi-Fi reliance heavily reduced your cellular data usage."
      Effect: NEGATIVE (green — a good thing for the user's data plan)

  ELIF wifi_ratio < 0.3:
      "Low Wi-Fi availability resulted in much higher cellular data dependence."
      Effect: POSITIVE (red — increases usage)

  IF gaming_hours > 2.0:
      "Active mobile gaming contributed significantly to the prediction."
      Effect: POSITIVE

  IF weekend_usage_ratio > 0.6:
      "Concentrating your data usage on weekends generally increases total
       monthly consumption due to binge habits."
      Effect: POSITIVE

  ELIF weekend_usage_ratio < 0.15:
      "Low weekend usage suggests you use data strictly for necessities,
       keeping overall consumption lower."
      Effect: NEGATIVE

  IF calls_made > 100:
      "High call volume indicates heavy phone usage but doesn't heavily
       impact cellular data."
      Effect: NEUTRAL

10.4 Baseline Comparison Tool (What-If Analysis)
--------------------------------------------------
A "Pin Baseline" feature allows users to lock in one prediction and compare
future predictions against it in real time:

  1. User runs first prediction: 5.2 GB → Clicks "Pin Baseline"
  2. User changes streaming_hours from 2 to 5 and re-runs
  3. New prediction: 8.7 GB
  4. System shows: "⬆️ +3.50 GB vs Baseline" (in red)

  Code:
    const diff = new_prediction - currentBaseline;
    if (diff > 0) → show red "⬆️ +X GB vs Baseline"
    if (diff < 0) → show green "⬇️ X GB vs Baseline"
    if (diff = 0) → show "➖ No change vs Baseline"

  This powerful UX feature lets users perform "what-if" experiments to
  understand how individual behaviour changes affect their data bill.
